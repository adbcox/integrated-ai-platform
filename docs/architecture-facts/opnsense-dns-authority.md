# OPNsense DNS state — UNDER INVESTIGATION

**Status:** PROVISIONAL. Asserted as fact in commit 237396b on
2026-05-01; that assertion was overconfident and is corrected here.
Definitive resolution: deliverable D-17-21.

## What is verified

- OPNsense at 192.168.10.1 is running OPNsense 26.1.6.
- Two DNS-related daemons are observable via API:
  - Unbound: running, answering on port 53, host overrides API
    returns 0 entries
  - Dnsmasq: running, answering on port 53053, hosts API returns
    6 entries (qnap, server2, ai-node-02, Mac-Mini-Eth,
    homeassistant, mac-studio — all bare hostnames with empty
    domain field)
- Several .internal records resolve via port 53 (grafana.internal,
  vault.internal, homepage.internal, qnap.internal among others —
  31 of 32 Caddy sites tested) despite no Unbound host_overrides
  and no Dnsmasq .internal hosts.
- Several .internal records do NOT resolve (mac-studio.internal,
  openproject.internal, docs.internal, netbox.internal among
  others) — including hosts that DO exist in Dnsmasq's Hosts table
  as bare names.

## What is suspected (not verified)

- Operator's intended DNS architecture is Dnsmasq + Kea (modern
  OPNsense 26.1 default), NOT Unbound.
- Unbound's daemon may have been enabled in a prior session by
  an earlier Claude/Aider/operator action that was never
  rolled back.
- The .internal resolution that DOES work may be coming from
  Unbound serving entries from an unintended source
  (autogenerated leases file, custom_options blob, or a config
  artifact left over from before the Dnsmasq migration).
- Per OPNsense 26.1 docs: when running Dnsmasq as resolver, "If
  you have Unbound DNS or any other DNS resolvers installed, make
  sure to disable them." The current state contradicts this
  guidance.
- Operator has seen this failure mode pre-Claude: two DNS
  services running simultaneously, port mapping breaks, requires
  full back-out of one to recover.

## What is NOT yet known

- Whether Unbound's daemon being active is genuinely unintended
  (most likely) or whether it was enabled deliberately for a
  reason that's been forgotten.
- Where the .internal records that DO resolve are actually being
  served from (config file path, OPNsense plugin, etc.).
- Whether the Caddy `tls internal` directive plays a role in
  registration of records.
- Whether a recent multi-day Vault troubleshooting incident on
  this platform had silent DNS-routing contribution that was
  misdiagnosed at the time. (Operator-flagged as worth checking.)

## What D-17-21 will resolve

The audit deliverable will:

1. Determine operator-intended DNS architecture by reviewing
   OPNsense GUI service list with operator
2. Verify Unbound's running state was unintended
3. Identify the source of currently-resolving .internal records
4. If Unbound is unintended: disable it cleanly via OPNsense's
   own service controls, after migrating its working records to
   Dnsmasq Hosts (or wherever the operator-intended authority
   lives)
5. Validate post-change: all currently-resolving records still
   resolve, no port collision, parity check accurately reports
6. Look back at recent troubleshooting incidents (notably the
   Vault network-routing incident) and assess whether DNS state
   could have contributed silently
7. Land a definitive replacement of THIS file with verified
   architecture facts

## How to verify (any future session must run this before any DNS work)

```
ROLE_ID=$(cat ~/.vault-approle/opnsense-api-reader/role-id)
SECRET_ID=$(cat ~/.vault-approle/opnsense-api-reader/secret-id)
LOGIN=$(curl -s -X POST -d "{\"role_id\":\"$ROLE_ID\",\"secret_id\":\"$SECRET_ID\"}" \
    http://127.0.0.1:8200/v1/auth/approle/login)
VT=$(echo "$LOGIN" | jq -r ".auth.client_token")
KEY=$(curl -s -H "X-Vault-Token: $VT" \
    "http://127.0.0.1:8200/v1/secret/data/opnsense/api" | jq -r ".data.data.api_key")
SEC=$(curl -s -H "X-Vault-Token: $VT" \
    "http://127.0.0.1:8200/v1/secret/data/opnsense/api" | jq -r ".data.data.api_secret")

# Count entries in each
echo "Unbound host overrides:"
curl -s -k -u "$KEY:$SEC" "https://192.168.10.1/api/unbound/settings/get" | \
    jq "[.unbound.hosts // {} | to_entries[] | select(.value.hostname != null and .value.hostname != \"\")] | length"

echo "Dnsmasq host entries:"
curl -s -k -u "$KEY:$SEC" "https://192.168.10.1/api/dnsmasq/settings/get" | \
    jq "[.dnsmasq.hosts // {} | to_entries[] | select(.value.host != null and .value.host != \"\")] | length"
```

Probe results are observable facts. The interpretation that one is
"the authority" is operator-intent, not probe-derivable. Both
probe non-zero is normal during a transition state. Wait for D-17-21
output before drawing authority conclusions.
