# Roadmap Intake and Normalization Process

## Purpose

This document defines how new roadmap inputs become canonical roadmap entries.

## Intake steps

1. capture the idea or request
2. determine whether it is architecture, roadmap, execution, or external-system material
3. assign or confirm canonical category and ID
4. normalize title, description, and scope
5. score priority, LOE, and architecture fit
6. identify dependencies, affected systems, and expected file families
7. evaluate grouped execution candidates
8. determine whether external documentation packs are required
9. update the canonical roadmap files

## Completion rule

An intake is not complete unless the canonical roadmap set reflects it.

## Anti-patterns

Do not leave new items only in:

- chat history
- temporary planning notes
- issues without normalized roadmap representation
- legacy directories not linked back into the canonical roadmap set
