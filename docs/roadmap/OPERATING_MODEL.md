# Roadmap Operating Model

## Purpose

This document explains how the roadmap should be operated as a managed planning system.

## Core operating model

- Architecture is the upstream design authority.
- Roadmap docs are the canonical planning and governance layer.
- GitHub issues, PRs, execution packs, and validation artifacts are downstream execution surfaces.
- Plane is the operational planning layer when used, but not the canonical planning source.

## Planning lifecycle

Roadmap items should move through a defined lifecycle:

1. intake
2. normalization
3. prioritization
4. grouping/dependency review
5. execution-readiness review
6. execution
7. validation
8. completion/archive synchronization

## Management rule

No roadmap item should move into active execution purely because it exists in backlog inventory form.
It must first satisfy readiness, dependency, and architecture-alignment expectations.

## PM shorthand

Treat the roadmap as a governed delivery system, not a note dump.
