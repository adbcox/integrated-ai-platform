## Section 2 — Probed capabilities (NOT documented)

### 2.1 — What sportarr is supposed to do

(From its container model and arr-stack pattern.) sportarr is a
DVR/automation tool for live sports: takes an event schedule (TV
channel + time + sport metadata), polls indexers for releases that
match, queues downloads through SABnzbd or torrent clients, and
files completed downloads into a media library.

### 2.2 — What sportarr is ACTUALLY doing right now

Logs over the last 24h (sample period) show:

```
[Indexer Status] Indexer 9 connection error #491:
                 Connection refused (localhost:9696)
[Indexer Status] Indexer 10 connection error #491:  ←
[Indexer Status] Indexer 11 connection error #489:  ←  All 5 indexers
[Indexer Status] Indexer 12 connection error #490:  ←  failing same way
[Indexer Status] Indexer 13 connection error #488:  ←
[Indexer Search] Fetched 0 total releases from 5 RSS feeds
[SABnzbd] Download SABnzbd_nzo_cfm48gsd failed:
          URL Fetching failed; Maximum retries
[DVR Auto-Scheduler] Scheduling complete - Checked: 3, Scheduled: 0,
                     Already Scheduled: 0, No Channel: 3, Errors: 0
```

Failure totals over the entire log (37,295 lines):

```
$ docker logs sportarr 2>&1 | grep -c "URL Fetching failed"
6769
```

So: container is "healthy" by Docker's healthcheck but operationally
producing zero output. Every cycle: 5 indexer fetches fail, 0
releases retrieved, 0 events scheduled (3 events queued have "No
Channel"), one stuck SABnzbd job retries and fails every 30s.

### 2.3 — Root-cause diagnosis

The indexer URLs in `/config/sportarr.db` (a SQLite store) all
point to `http://localhost:9696/...`:

```
showRSS (Prowlarr)    http://localhost:9696/2/<apikey>/api
1337x (Prowlarr)      http://localhost:9696/1/<apikey>/api
TorrentDay (Prowlarr) http://localhost:9696/4/<apikey>/api
NZBgeek (Prowlarr)    http://localhost:9696/5/<apikey>/api
EZTV (Prowlarr)       http://localhost:9696/3/<apikey>/api
```

Inside the sportarr container, `localhost:9696` is sportarr's own
loopback — not Prowlarr's. Prowlarr **is** running (`docker ps`
shows it on host port `9696`), reachable from sportarr at
`prowlarr:9696` over the `control-center-net` bridge.

The failed SABnzbd downloads are downstream of the same bug —
sportarr asks Prowlarr (at the wrong URL) for the actual download
URL, never gets one, retries forever.

The "No Channel" issue (3 monitored events, 0 schedulable) is a
separate misconfiguration — even if indexers worked, sportarr
wouldn't schedule because it has no channel mapping for the events
it knows about.

### 2.4 — Resource-cost interpretation

394 MiB resident + 35 PIDs + 948 MB net rx in 24h, all producing
zero output. The 948 MB is mostly the retry traffic to sister
arr-services (rejected on `localhost:9696`) plus log growth.

### 2.5 — Available but not exercised

- **Live event scheduling** — supported but blocked by "No Channel"
- **Download history / library file management** — likely supported;
  irrelevant when no downloads succeed
- **API surface on `:1867`** — not probed; orthogonal to whether
  the tool's core function works

## Section 3 — Stack-coverage check

For each probed capability:

### 3.1 — Live sports DVR scheduling

- **What other tool covers this?** None in the platform stack.
  - Sonarr: TV series automation (episode-based, not event-based)
  - Radarr: movies (one-shot releases, not scheduled events)
  - Prowlarr: indexer aggregation, doesn't schedule
- **Verdict:** UNIQUE in the stack — IF the operator wants live
  sports automation. If the operator does NOT want live sports
  content automated, then "no other tool covers this" doesn't
  matter; the capability is unused.

### 3.2 — Indexer aggregation

- **What other tool covers this?** Prowlarr is the indexer
  aggregator. Sportarr CONSUMES Prowlarr; doesn't replicate it.
- **Verdict:** Sportarr does not duplicate Prowlarr.

### 3.3 — Download client integration

- **What other tool covers this?** Sonarr and Radarr both speak to
  SABnzbd already. Sportarr would too if its requests got through.
- **Verdict:** Pattern duplicated, but each *arr instance owns its
  own indexer→download queue lifecycle.

The stack-coverage answer here is conditional on operator intent
(see §3.1). That's why this audit cannot decide unilaterally — the
verdict requires the operator's preference.

## Section 4 — Verdict (operator decision required)

This is a **fix-or-retire** verdict per the D-17-02 template's
extension for broken services. Three options, each with the explicit
trade-off:

### Option A — FIX (~30-60 min)

Bounded scope:
1. Patch the 5 indexer rows in `/config/sportarr.db` to use
   `http://prowlarr:9696/...` instead of `localhost:9696/...`
   (sqlite UPDATE).
2. Restart sportarr; verify indexer status shows green; verify the
   stuck SABnzbd job either completes or fails cleanly.
3. **Separate from the indexer fix:** investigate the 3 "No
   Channel" monitored events. Either map channels (operator-
   knowledge work) or delete the orphan events.

Choose this only if the operator currently wants live-sports
automation. The 6,769 failed downloads in the log suggest this has
been broken for weeks/months — if the operator hasn't missed it,
that's evidence the capability isn't load-bearing.

### Option B — PARK-RETIRE (~30 min)

1. `docker compose -f /Users/admin/control-center-stack/stacks/arr-stack/docker-compose.yml stop sportarr`
2. Remove sportarr block from compose.yml (operator-confirmed
   in-place edit, since the compose is out-of-repo).
3. Snapshot the compose change in
   `docs/_retired/sportarr-2026-05-01.md` (or equivalent; the
   in-repo `docker/_retired/` pattern from D-17-05 doesn't apply
   directly because the compose file is out-of-repo).
4. Keep the `sportarr-config` Docker volume so a future operator
   can revive without losing indexer config + DVR history.
5. Remove from any dashboards (homepage/homarr) that reference it.

Choose this if operator does not currently want live-sports
content. Easily reversed.

### Option C — KEEP WITH RECONFIG REQUIRED (defer to dedicated deliverable)

If operator wants live-sports AND the fix is non-trivial (e.g.,
the indexer URL fix won't solve the "No Channel" problem and the
operator wants a clean reset), promote to a dedicated deliverable
later in Phase 17.

In that case, D-17-08's commit captures the audit + the deferral
decision; no container change.
