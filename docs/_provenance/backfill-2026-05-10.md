## Method

Per Path A doctrine in `docs/architecture-facts/model-provenance.md`
(§"Backfill verdicts"), backfill verdicts use one of three sources,
named explicitly per row:

- **scan-confirmed:** `scripts/verify-model-provenance.sh` was run
  against the HF source repo; verdict + top-K captured in
  `docs/_provenance/<sanitized>.json`.
- **catalog-inferred:** T0.5 catalog enumeration confirmed the HF
  source's presence (or absence) in the deep-signals fingerprint DB;
  verdict deduced from catalog presence + variant-scoring caveat.
- **mapping-inferred:** Local artifact → HF source repo is
  deterministic per mapping config; verdict inferred from the mapped
  HF repo's catalog status.

This backfill introduces a fourth, **operator-accepted disposition**,
applied where scan-confirmed pursuit is blocked by hardware constraint
and the operator accepts the next-best evidence class as the active
doctrine class. See "Operator decisions surfaced" below.

## Operator decisions surfaced

**Path B — operator-accepted SCAN_OOM disposition (2026-05-10):**

The upstream Qwen scan failed on both available local hosts due to
unified-memory exhaustion during BF16 fingerprint computation. The
operator chose Path B from three options surfaced at the WP-03
hard stop:

- **(A)** Run scan on Mac Studio (96 GB unified memory) — deferred
  until Headscale connectivity is restored; tracked as KI-010.
- **(B)** Accept scan-failed/SCAN_OOM as a documented disposition
  that falls back to the existing `hf_id_confidence: operator_confirmed`
  doctrine class. **Selected.**
- **(C)** Run with `PROVENANCE_FORCE=1` after expanding swap — not
  pursued; prior 9564s runtime suggests no swap-mediated path completes
  on 32 GB hardware.

Path B rationale: the upstream `Qwen/Qwen3-Coder-30B-A3B-Instruct` HF
ID was already operator-confirmed on 2026-05-05 (see
`config/model_provenance/ollama_to_hf_mapping.yaml` line 23). The
Cisco kit scan would have promoted that confidence to `verified-*`,
but operator-confirmed is the doctrine-acknowledged next-best class
when scan is hardware-blocked. The disposition reflects the actual
evidentiary state: HF ID matched-by-architecture (qwen3moe + 30.5B
total / 3B active + 262K context), not matched-by-fingerprint.

**Operational vs governance distinction (important for future readers):**

The SCAN_OOM is specific to the BF16 fingerprinting workload required
by the Cisco kit. It is **not** a constraint on the operational use
of the MLX-3bit serving model. The MLX-3bit asset is 12.5 GiB on
disk, runs cleanly on macbook-pro 32 GB (proven 2026-05-10
integration test: 368s wall-clock for TASK-0001, 21.5% improvement
over Ollama baseline; vllm-mlx process RSS 1.7 GB with weights
resident in Metal GPU heap). Serving operates well within hardware;
governance attestation is what's blocked.

The MLX-3bit derived disposition is therefore
**quantization-of-operator-confirmed-base**, distinguishable from
**quantization-of-verified-base** (the doctrine class that would apply
post-Mac-Studio rescan, per D-17-92 quantization-divergence doctrine).
