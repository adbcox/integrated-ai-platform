# Current State vs Target State

## Purpose

This document summarizes the gap between the current platform state and the target architecture.

## Current state

The current platform has:

- a meaningful control-plane foundation
- orchestration and budget logic
- promotion/qualification surfaces
- benchmark and learning-loop artifacts
- growing roadmap and governance discipline
- increasing clarity around adopted external systems

But it still lacks a fully normalized shared runtime substrate.

## Target state

The target state is a platform with:

- retained and hardened control plane
- implemented shared runtime substrate
- stable inference gateway with Ollama-first posture
- typed tools and governed workspaces
- permission and sandbox enforcement
- consistent artifact outputs
- repeatable validation and promotion evidence
- controlled branch expansion on top of the same substrate

## Most important delta

The biggest delta is the missing shared runtime substrate.
That remains the critical architectural gap.

## Secondary deltas

Additional important deltas include:

- stronger authority-surface normalization
- fuller telemetry/evidence completeness
- cleaner external-system integration discipline
- broader branch expansion only after substrate maturity
- more complete CMDB/inventory alignment over time

## Practical implication

Roadmap and execution work should continue to prefer changes that move the platform from the current state toward the target state rather than broadening surface area without substrate progress.
