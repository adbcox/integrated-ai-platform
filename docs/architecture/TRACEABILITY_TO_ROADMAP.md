# Architecture to Roadmap Traceability

## Purpose

This document provides a lightweight traceability view from architecture priorities to roadmap families and governance surfaces.

## Core architecture priorities

### Shared runtime and local autonomy
Supported most directly by:
- `RM-AUTO-001`
- `RM-DEV-001`
- `RM-OPS-004`
- `RM-OPS-005`

### Governance, anti-drift, and planning integrity
Supported most directly by:
- `RM-GOV-001`
- `RM-GOV-002`
- `RM-GOV-003`
- `RM-GOV-006`
- `RM-GOV-007`
- `RM-GOV-008`

### Inventory and capability understanding
Supported most directly by:
- `RM-INV-001`
- `RM-INV-002`
- `RM-INV-003`

### Controlled domain expansion
Supported by domain families including:
- `RM-HOME-*`
- `RM-MEDIA-*`
- `RM-SHOP-*`
- `RM-AUTO-MECH-*`
- `RM-LANG-*`
- `RM-HW-*`
- `RM-DOCAPP-*`

## Practical rule

If a roadmap item cannot be explained in terms of one of the architecture priorities above, it likely needs to be re-scoped, decomposed, or delayed.
