## Finding B — Sportarr missing from Caddy routes

**Severity:** P3 (functional gap, not a regression — service was not
Caddy-fronted at deployment)

Sportarr (port 1867) is running and healthy but has no Caddy reverse
proxy route in `docker/caddy/Caddyfile` and no `sportarr.internal`
Dnsmasq host override. Access is currently via direct LAN IP + port.

This was not a gap introduced by this audit period — sportarr was
originally deployed direct-port (D-17-36). The absence is deliberate
but undocumented.

**Disposition:** Document as Known Limitation. Adding a Caddy front for
sportarr is a separate deliverable (requires Dnsmasq host entry +
Caddyfile entry + `caddy-dns-parity --refresh`). Not blocking this
audit.

## Finding C — Original services (sonarr, radarr, prowlarr, sportarr) lack provision scripts

**Severity:** P4 (informational, pre-dates the D-17-47 provision-script
pattern)

Services deployed before D-17-47 were bootstrapped via manual/ad-hoc
credential management and do not have `scripts/provision-<svc>.sh`
companions. The provision-script pattern was introduced with Bazarr
(D-17-47) as the canonical shape.

| Service | provision script | vault-agent dir | policy hcl |
|---|---|---|---|
| sonarr | MISSING (pre-pattern) | MISSING (Vault reads by buildarr/scraparr sidecars) | sonarr-policy.hcl ✓ |
| radarr | MISSING (pre-pattern) | MISSING (same) | radarr-policy.hcl ✓ |
| prowlarr | MISSING (pre-pattern) | MISSING (same) | prowlarr-policy.hcl ✓ |
| sportarr | MISSING (pre-pattern) | MISSING (no Vault sidecar) | MISSING |
| bazarr | provision-bazarr.sh ✓ | vault-agent-bazarr/ ✓ | bazarr-policy.hcl ✓ |
| cleanuparr | provision-cleanuparr.sh ✓ | vault-agent-cleanuparr/ ✓ | cleanuparr-policy.hcl ✓ |
| lidarr | provision-lidarr.sh ✓ | vault-agent-lidarr/ ✓ | lidarr-policy.hcl ✓ |

Note: Sonarr/Radarr/Prowlarr API keys reach those services via their
own internal config.xml, not via a vault-agent sidecar pattern. The
pattern gap is architectural (original arr-stack predates the sidecar
doctrine) and is not a credential-availability issue.

Sportarr has no Vault policy because it generates no secrets and reads
none from Vault. Policy is genuinely N/A.

**Disposition:** Document as known architecture-generation gap.
Retrofitting provision scripts for pre-pattern services is a future
deliverable if operator wishes to standardize. Not blocking.

## Finding D — D-17-87 `--no-verify` retroactive note

**Severity:** P3 (process deviation, acknowledged)

D-17-87 (Lidarr deployment) was committed with `--no-verify`, bypassing
the `detect-secrets` pre-commit hook. This is acknowledged as a
retroactive process deviation. No secrets were in the diff (the
`--no-verify` was used to bypass a hook unrelated to credential
content). The commit is safe but violated hook discipline.

Chronicled here and in D-17-88 closure note in PROJECT_FRAMEWORK.md.
No remediation required beyond documentation; future sessions should
use `--no-verify` only when the specific hook being bypassed is
verified safe.
