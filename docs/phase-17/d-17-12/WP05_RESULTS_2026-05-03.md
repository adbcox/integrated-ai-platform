## Headline findings

### F1. T3-B qwen3-coder:30b emits STRUCTURED tool_calls

**Major divergence from qwen2.5-coder family.** Every tool-call task on T3-B
produced `response.message.tool_calls` populated by Ollama (not inline JSON
text in `response.message.content`). T3-B passed all 6 tool-call tasks at
score 0.93 mean.

This **flips the local-tool-calling.md Finding 1 conclusion** for the qwen3
family: it is NOT a generic Ollama-serving-stack issue. It is qwen2.5-coder-
specific. Significance for the platform:

- D-17-13 Goose blocker re-evaluation: the right local model for Goose is
  qwen3-coder:30b, not qwen2.5-coder:* — Goose can speak structured tools
  natively against T3-B without inline-JSON adapters.
- The model-side finding extends Finding 1 from "Ollama serving issue" to
  "qwen2.5-coder model-side; qwen3-coder structured-correct." This is a
  concrete contribution back to local-tool-calling.md doctrine.

### F2. Gemma 2 cannot tool-call

`gemma2:27b` returns HTTP 400 from Ollama: `"registry.ollama.ai/library/
gemma2:27b does not support tools"`. Architecture-level limitation, not
serving stack. T3-A tool-call cell is correctly recorded as `n/a` (n_error=6
status=error) — comparable to T3-A long-context cell where 4/5 tasks were
n/a-context-insufficient.

## Auto-grading caveats already known to fire

### G1. refactor-add-logging false-fails ALL models

Every model scored exactly 0.50 with grader notes "3/3 required transforms; 1
forbidden". The forbidden item is `print(`. The task input includes this
existing caller in `file_d.py`:

```python
# file_d.py — caller
from file_c import greet
print(greet(42))
print(greet(99))
```

Models correctly preserve the caller (per task instruction "do not change
public API"); grader penalizes them for it. **Task-design bug, not model
defect.** Rerunning with `must_not_contain` removed (or scoped to non-caller
files) would lift all four models' refactor mean by ~0.12.

For WP-07, treat refactor cell scores as `score + 0.12 / 4 ≈ +0.03 mean
correction`. Small impact on rank-stability.

### G2. tool-call grader gives 0.45–0.60 for inline-JSON correctness

T1 + T2 inline-JSON emissions score 0.45 base + correctness bonuses, while
T3-B structured emission scores 0.60 base + bonuses. Two scoring tracks for
the same observable behavior. Worked example:

| Model | tc-search-event | Note |
|---|---|---|
| T1 (inline JSON) | 0.45 | inline-correct, no structured |
| T2 (inline JSON) | 0.45 | inline-correct, no structured |
| T3-B (structured) | 0.60 | structured-correct |

The 0.15 gap is grader policy not model gap. WP-07 should:
(a) report the structured-only pass rate separately, OR
(b) collapse inline+structured to a single track and score on correctness only.

### G3. long-context keyword-grading

Auto-grader counts case-insensitive substring hits in `expected_facts`. Risk:
model "spits keywords" without coherent reasoning. Suggested 2 hand-grades
per (model × long-context cell) — see "Hand-grade sample" below.

## Mid-run drift recorded

**Mac Studio Ollama bind misconfiguration** discovered after primary run
started. Brew-managed launchd plist had no `OLLAMA_HOST`, defaulting to
`127.0.0.1:11434` — unreachable from Mac Mini harness. Fixed mid-run by:

1. Stopping `brew services stop ollama` (which deleted the brew-managed plist)
2. Starting `OLLAMA_HOST=0.0.0.0:11434 OLLAMA_FLASH_ATTENTION=1
   OLLAMA_KV_CACHE_TYPE=q8_0 nohup ollama serve` (PID 71160 on Studio)
3. Verifying bind: `lsof -nP -iTCP:11434 -sTCP:LISTEN` → `*:11434`
4. Verifying reachability: `curl http://192.168.10.142:11434/api/tags` from Mini

Studio Ollama daemon is now nohup-managed (will not survive Studio reboot).
For benchmark continuity within D-17-12 this is fine. For "future-reuse asset"
(per operator note: Studio is a permanent compute substrate, not a one-off),
WP-08 should record a follow-on for proper launchd registration with `OLLAMA_
HOST=0.0.0.0:11434` baked in. This is a **second instance** of the launchctl/
plist pattern after Mac Mini Finding Y — both hosts now have nohup workarounds
where launchd should own the daemon.
