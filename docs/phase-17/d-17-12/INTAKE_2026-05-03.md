### Tool-call adherence workload — scope limitation

Per `docs/architecture-facts/local-tool-calling.md` Findings 1+2:

- Ollama drops `tool_calls` in streaming mode.
- exo's OpenAI-compat shim returns Qwen `<tools>` markup as text.

Both are upstream-blocked. Benchmarking tool-call adherence via a
streaming agent host would measure **the serving stack's tool-call
protocol fidelity, not the model's tool-call competence.** Scope
restricted: `stream:false` direct-curl only. This measures *model
competence in isolation*; *agent-integration* tool-call performance
is deferred until Findings 1+2 unblock. Limitation called out
explicitly in the WP-08 chronicle.

## Workload definitions

Each model runs all 4:

1. **Long-context reasoning.** 16K+ token inputs (held-out
   technical documents, multi-section analysis tasks). Tests
   whether T3 candidates exploit the larger context profitably or
   simply soak more memory for no quality win.
2. **Multi-file code refactor.** 5+ source files, cross-reference
   required. Tests Qwen3-Coder's specialty thesis.
3. **Tool-call adherence (`stream:false` only).** Direct-curl to
   `/v1/chat/completions` with `tools=[...]`. Score: did the model
   emit a structured `tool_calls` array, did the args parse as the
   declared schema.
4. **Autonomous agentic.** Multi-step task with verify loops.
   Score: did the model follow the plan, did it self-correct when
   verification said "no", did it terminate when complete.

## Win conditions

Per workload × model pair:

- **Quality:** human-graded correctness on held-out task set.
  Sample size pinned at WP-06 surface-back.
- **Latency:** time-to-first-token + total time per task.
- **Throughput:** tokens/sec sustained over the response.
- **Memory pressure:** peak unified memory occupancy from Ollama
  runner subprocess RSS.

Decision matrix at WP-07: **adopt iff ≥2 quality dimensions clear
T1/T2 on the model's specialty workload AND latency/memory
acceptable.**
