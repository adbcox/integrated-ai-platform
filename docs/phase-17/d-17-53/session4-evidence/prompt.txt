============ THE TASK ============

Author a draft of:
  docs/architecture-facts/vault-agent-sidecar-pattern.md

Goal: Generalized canonical pattern doc for Vault Agent sidecar
credential delivery on this platform. Abstracted from the N=5
worked examples (dashboard, buildarr, bazarr, scraparr, cleanuparr).
Future arr-stack expansions and any new credential-consuming
service will reference THIS file as the canonical pattern.

This is doctrine, not a runbook. Audience is a future engineer
who needs to understand the pattern well enough to apply it
correctly to a new service.

Required coverage (each section short and grounded in the worked
examples — no padding):

1. **Pattern overview.** What problem this pattern solves:
   credentials never live in filesystem-tracked files, never in
   repo, never in chat output. They reach the consumer at runtime
   via Vault Agent rendering to a tmpfs-backed env file that
   the consumer mounts read-only.

2. **Architecture diagram (ASCII / text).** The data flow:
   AppRole role-id + secret-id provisioned at deploy time →
   Vault Agent sidecar authenticates with AppRole → Vault Agent
   reads policy-scoped paths → Vault Agent renders template to
   credentials.env file on a shared volume → consumer container
   sources credentials.env at startup. One diagram, no padding.

3. **Required artifacts per service.** What files / config
   exist for each new service that adopts this pattern.
   Extract the recurring shape:
   - `config/vault-policies/<service>-policy.hcl` — what
     paths and capabilities the policy grants. Note the
     pattern of read-only on `secret/data/<service>/*` paths
     and any platform-shared paths.
   - `docker/vault-agent-<service>/agent.hcl` — agent config
     shape: AppRole auto-auth method, sink, template stanza
     pointing at credentials.env.tmpl. Note any recurring
     fields and any per-service variation.
   - `docker/vault-agent-<service>/credentials.env.tmpl` —
     template shape (Vault `template { contents = "..." }`
     style with `{{ with secret "..." }}` blocks). Describe
     the SHAPE only — do not list values, do not list paths
     that imply credential structure.
   - `scripts/provision-<service>.sh` — what the provision
     script does (load admin token via shared lib, create
     policy, create AppRole, render role-id + secret-id to
     local files Vault Agent reads at boot).
   - `docker-compose` service entry — sidecar container,
     consumer container, shared volume, dependency / startup
     ordering.

4. **AppRole bootstrap flow.** How `scripts/lib/vault-admin-
   token.sh` is consumed by every provision script — its role
   in the bootstrap, what it returns, and why provision scripts
   never carry the admin token themselves.

5. **Hash-only verification doctrine.** Reference D-17-38:
   credential equality is verified via `sha256[:12]` fingerprint
   comparison, never by displaying values. Cite the
   `/proc/1/environ` redacted-grep pattern (D-17-38 Finding 6
   sub-doctrine) as the correct way to check that a credential
   variable is *present* in a running container without leaking
   the value.

6. **Failure modes observed across N=5 examples.** Look for
   recurring failure modes referenced in the worked examples'
   commit messages or surrounding doctrine. If the worked
   examples surface specific failure modes (e.g. URL drift,
   template-rendering race, sidecar startup ordering), capture
   them generically. If you cannot observe failure modes from
   the source files alone, mark
   `[UNVERIFIED — frontier review]` and request frontier
   correction with cross-deliverable context.

7. **When NOT to use this pattern.** Cases the pattern does not
   fit: services that don't consume credentials (frontend-only
   like plane-web), services managed outside compose (Obot-
   managed shims per CLAUDE.md D#30), services that read
   credentials from a different substrate (LLM-routing tokens
   that flow through litellm, etc.). Be conservative — if you
   are not sure, mark [UNVERIFIED] and surface back rather than
   speculating.
