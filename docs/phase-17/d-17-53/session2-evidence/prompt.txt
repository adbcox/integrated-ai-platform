============ TASK A ============

Author a draft of:
  docs/runbooks/opnsense-dhcp-dns-push.md

Goal: Document the procedure to configure OPNsense DHCP to push
192.168.10.1 (Dnsmasq) as primary DNS server to LAN clients. This
closes the gap where MacBook + new clients query public DNS instead
of the internal Dnsmasq authority, breaking .internal resolution
without manual DNS config.

Required coverage:
- Why this is needed. Sibling to D-17-21 DNS authority work
  (Dnsmasq is sole authority for *.internal). Reinforced by
  Finding 14 (consumer-side cache invalidation) — pushing the
  correct DNS server at lease time prevents a class of stale-cache
  issues at first connect.
- OPNsense UI navigation path. Verify which surface configures
  this from your reading: Services → Dnsmasq DNS → Settings, or
  Services → ISC DHCPv4, or Services → Kea DHCPv4. (Note: at
  D-17-21 the platform stated "Dnsmasq + Kea per OPNsense 26.1
  modern defaults" — Kea is the current DHCP server, not ISC.
  But the DNS-push option may live under either Dnsmasq's DHCP
  bridge or the Kea config UI. If you cannot determine the exact
  UI path from the source docs, flag [UNVERIFIED — frontier
  review].)
- Config option / UI field. The dhcp-option=6,192.168.10.1
  pattern is the dnsmasq-config form; the OPNsense UI typically
  exposes this as "DNS servers" field on the DHCP scope/subnet.
  Document both.
- Verification steps. DHCP lease renewal on client, observe DNS
  server pushed (e.g. macOS: `scutil --dns | grep nameserver`,
  Linux: `resolvectl status`), dig test against pushed server.
  Note: you cannot run these yourself; document as operator
  steps.
- Rollback path. What field/setting to revert if the push breaks
  client connectivity.
