============ THE TASK ============

Author a draft of:
  docs/runbooks/opnsense-dhcp-dns-push.md

**Goal:** Configure OPNsense Dnsmasq DHCP to push 192.168.10.1
(OPNsense LAN IP, where Dnsmasq runs) as primary DNS server to LAN
clients. Closes the gap where new clients query public DNS instead
of internal Dnsmasq, breaking `.internal` resolution without manual
per-client DNS config.

This is a runbook, not doctrine. Audience: operator (or a future
Goose session) walking through configuring DHCP-DNS-push on the
existing Dnsmasq instance.

Required coverage:

1. **Why needed.** Sibling to D-17-21 (DNS authority) and Finding 14
   (cache invalidation). Without DHCP-DNS-push, a fresh DHCP client
   gets whatever DNS its OS defaults to — typically the gateway
   relaying upstream public DNS — and `.internal` records won't
   resolve until the operator hand-configures the resolver.

2. **OPNsense UI navigation.** Services → Dnsmasq DNS → [DHCP
   options section]. Verify exact field name / nav path against
   `opnsense-add-host-overrides.md` style and the OPNsense
   Dnsmasq pattern documented in `opnsense-dns-authority.md`. If
   you can't verify a UI label from source files, mark it
   "[UNVERIFIED — frontier review]" inline.

3. **Config option.** Dnsmasq native syntax:
   `dhcp-option=6,192.168.10.1` (option 6 = DNS server). Document
   the UI-equivalent field if reachable from source files; mark
   [UNVERIFIED] if not.

4. **Prerequisite check.** Verify Dnsmasq is functioning as DNS
   authority before pushing it as DHCP option (per
   `opnsense-dns-authority.md`). If Dnsmasq DNS is broken, pushing
   it as the DHCP DNS option breaks every fresh DHCP lease.

5. **Verification.** DHCP lease renewal on a client + DNS check:
   - macOS: `scutil --dns` (look for resolver entries) +
     `dig <something>.internal`
   - Linux: `resolvectl status` + `dig <something>.internal`
   - Show expected output shape, not a specific record (records
     change; the shape is stable)

6. **Rollback.** Revert the DHCP option setting + cache flush per
   Finding 14 (mDNSResponder flush on macOS; equivalent on
   Linux). Note that clients with already-leased addresses will
   not pick up the rollback until lease renewal or manual
   `dhclient -r` / network-toggle.
