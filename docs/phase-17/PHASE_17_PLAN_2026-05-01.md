| red | >70% on either OR `docker_status != running` |

as primary visual signal.

**T5 — Drift gating**

**T8 — Retroactive incident review: Vault troubleshooting hypothesis**

**Operator hypothesis (control window, 2026-05-01):**

The multi-day Vault troubleshooting incident that was diagnosed as
"Vault internal failure + network routing" may have actually been a
single root cause: DNS resolution latency through Unbound's
unintended-active state.

**Symptom pattern operator observed:**

- Vault unseal would fail consistently on the third Shamir key
  - Hypothesis: cumulative DNS lookup latency for vault.internal
    across the multi-step unseal flow exceeded Vault's per-operation
    timeout. Failure mode looked like Vault internal error; actual
    cause was external DNS delay
- Sonarr and Radarr remained "functional but slow" during the same
  period
  - Hypothesis: timeout-tolerant clients (queue/retry-based) absorbed
    the DNS latency and presented as degraded-but-working;
    timeout-strict clients (Vault unseal) failed hard
- IP+port direct access to .145 worked (slowly) when .internal did
  not
  - Hypothesis: bypassing DNS resolution entirely avoided the latency
    path; the delay was specifically in hostname resolution, not in
    underlying network routing
- The "fix" that resolved the incident was characterized as network
  routing, but the speed improvement post-fix suggests DNS resolution
  path was unblocked
  - Hypothesis: whatever the "network routing" fix did, it
    incidentally bypassed or sped up the slow-DNS path; the
    documented diagnosis attributed the wrong proximate cause
- Operator raised the DNS connection during incident; was told
  "those are separate issues." They were not, in this hypothesis.

**Why this hypothesis matters beyond historical accuracy:**

- If correct: any future incident with timeout-sensitive services
  on .internal hostnames is at risk of the same misdiagnosis
- The dismissal pattern ("operator raised it, Claude said separate")
  is the same anti-pattern that produced the Unbound architecture-
  fact mistake. Capturing this incident postmortem reinforces D#22
  (operator pushback is data, not noise) with a concrete worked
  example.
- D-17-21's success criteria includes preventing the same failure
  mode from recurring. If the DNS-state fix in T1-T6 closes the
  hypothesized contributor, validation should include re-running
  Vault unseal flow under load and confirming consistent
  third-key success.

**Investigation actions:**

a. Find the Vault troubleshooting incident postmortem/runbook (likely
   in docs/runbooks/ or docs/incidents/). Quote the documented
   diagnosis verbatim. Quote operator pushback messages from the
   relevant chat transcript if accessible via conversation_search.

b. Identify timeline: when did Unbound first start answering port 53
   on this OPNsense? Compare to when Vault troubleshooting incident
   started. If Unbound enable-time predates incident start, that's
   evidence of contribution.

c. After T5 (Unbound disabled), measure DNS resolution latency for
   vault.internal at port 53 vs current state. If pre-fix latency
   was meaningfully higher (10ms+ matters for a multi-step flow),

d. Run a Vault unseal cycle post-T5 and confirm no third-key timeout.

e. Annotate (do NOT rewrite) the original incident postmortem with:
