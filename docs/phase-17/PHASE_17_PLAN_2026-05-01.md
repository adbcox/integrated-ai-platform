This plan does NOT execute any Tier 1-4 deliverables. Each is its own deliverable prompt, scoped after this plan lands and is operator-reviewed.

### 17.Q — Logical service architecture dashboard with live utilization

Replaces original "architecture dashboard on OpenProject" framing
(scoping error: OpenProject is project tracking, not visualization).

Substrate: extend existing topology-api (port 8300) which already
serves `/api/topology` in Grafana Node Graph format with 75 nodes +
72 edges and per-node `detail__purpose` descriptions.

**Deliverables:**

**T1 — topology-api enrichment**
- Pull `detail__purpose` dynamically from xindex `/service/{name}`
  instead of hardcoded source strings; this means description edits
  flow through xindex's existing knowledge layer.
- Add node fields:
  - `utilization_cpu_pct` — last-5m avg from VictoriaMetrics
  - `utilization_memory_pct` — current vs container limit
  - `utilization_traffic` — request rate or bytes/sec where a
    meaningful proxy exists
  - `docker_status` — running / restarting / unhealthy
  - `docker_restart_count` — last 24h restart count
  - `docker_last_log_error` — most recent ERROR/FATAL log line
    (truncated to 200 chars)

**T2 — Color-state derivation**

Replace category-based color (current behavior) with utilization-
pressure color. Suggested band:

| Color | Condition |
|---|---|
| green | `utilization_cpu_pct < 30` AND `memory_pct < 50` |
| yellow | 30-70% on either dimension |
| red | >70% on either OR `docker_status != running` |
| grey | no metrics available (e.g., service that doesn't expose container metrics) |

Category info preserved as the subtitle field (already present), not
as primary visual signal.

**T3 — Grafana Service Topology dashboard**
- Verify existing dashboard or create new one. JSON model in
  `config/grafana/dashboards/service-topology.json`.
- Uses Node Graph panel via `marcusolsson-json-datasource` pointed
  at topology-api `/api/topology`.
- Hint text in tooltip from `detail__purpose`. Refresh cadence ~30s.

**T4 — Health-and-utilization overlay panel**

Companion panel on same dashboard: per-service utilization
sparklines, restart-count counter, and docker status indicators.
Cross-references the Node Graph (clicking a node filters the
overlay panel).

**T5 — Drift gating**

topology-api's existing pre-commit lint stays. Add: if a service is
in NetBox but not in topology-api's response, fail. If a node has
no xindex `/service/{name}` match, fail (prevents drift between
ingested knowledge and rendered topology).

**Estimate:** 12 hours.

**Dependencies:** requires 17.F closed (so retired services are out
of topology-api before utilization wiring; otherwise wasted work).
No OpenProject dependency.

### 17.R — Physical architecture visualization

Replaces original "Structurizr population" framing — that becomes T1
inside this deliverable, not the whole deliverable.

Companion to 17.Q's logical view. Same data sources (NetBox + xindex
+ VictoriaMetrics) viewed differently: nodes-on-hosts, network paths
(Ethernet links / TB5 future link), storage paths (QNAP SMB, Vault,
Restic backup chain).

**Deliverables:**

**T1 — Structurizr population**

Existing Structurizr container is running (container ID
`16af27248e96` per probe). Determine population state. If empty,
populate with platform model:

| Element | Source |
|---|---|
| SoftwareSystem | Integrated AI Platform |
| Containers | per NetBox device list (5 currently: mac-mini, mac-studio, opnsense, qnap, ha) |
| Components | per xindex `/service` ingest (75 services) |
| Relationships | per topology-api edges (72 dependencies) |

Workspace DSL committed to `config/structurizr/workspace.dsl`;
rebuild on commit triggers Structurizr container reload.

**T2 — Decision: Structurizr OR custom Grafana panel OR static SVG**

Evaluate each:

| Option | Best for | Weakness |
|---|---|---|
| Structurizr | layered C4 view | weakest for live utilization data (design-time, not runtime) |
| Custom Grafana panel | live data overlay | requires custom panel plugin or Mermaid+templating |
| Static SVG | simplicity (regenerated nightly from NetBox+xindex) | no live overlay |

Pick one for the primary surface. The other two may also exist as
supplementary views if cheap.

**T3 — Network path overlay**
- Mac Mini ↔ Mac Studio: show 1 GbE current link, future TB5
  120 Gb/s link (post-17.N) as alternative path
- Mac Mini ↔ QNAP: 1 GbE
- Mac Mini ↔ OPNsense: 1 GbE
- Where VictoriaMetrics has link utilization data (network byte
  counters), surface as live link load %

**T4 — Storage path overlay**
- Restic backup chain: Mac Mini `/vault/data` → Mac Mini Restic
  client → MinIO@QNAP → QNAP disks
- QNAP SMB share → Mac Mini `/Users/admin/mnt/qnap-downloads`
- Vault data path on Mac Mini

**T5 — Capability ceiling annotations**

Per node, show "currently using X% of capability ceiling" pulled
from the 17.C audit's capability sections combined with current
utilization. Mac Studio: shows 96 GB ceiling, near-zero usage
currently. Mac Mini: shows 48 GB ceiling, current usage X%.

**Estimate:** 8 hours.

**Dependencies:** 17.Q closed (T2 decision benefits from seeing the
logical view's live-data approach first). Optional dependency on
17.O Day-1 for Mac Studio physical state (already complete).
