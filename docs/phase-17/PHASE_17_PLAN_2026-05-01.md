### 17.Q — Logical service architecture dashboard with live utilization

Replaces original "architecture dashboard on OpenProject" framing
(scoping error: OpenProject is project tracking, not visualization).

Substrate: extend existing topology-api (port 8300) which already
serves `/api/topology` in Grafana Node Graph format with 75 nodes +
72 edges and per-node `detail__purpose` descriptions.

**Deliverables:**

**T1 — topology-api enrichment**
- Pull `detail__purpose` dynamically from xindex `/service/{name}`
  instead of hardcoded source strings; this means description edits
  flow through xindex's existing knowledge layer.
- Add node fields:
  - `utilization_cpu_pct` — last-5m avg from VictoriaMetrics
  - `utilization_memory_pct` — current vs container limit
  - `utilization_traffic` — request rate or bytes/sec where a
    meaningful proxy exists
  - `docker_status` — running / restarting / unhealthy
  - `docker_restart_count` — last 24h restart count
  - `docker_last_log_error` — most recent ERROR/FATAL log line
    (truncated to 200 chars)

**T2 — Color-state derivation**

Replace category-based color (current behavior) with utilization-
pressure color. Suggested band:

| Color | Condition |
|---|---|
| green | `utilization_cpu_pct < 30` AND `memory_pct < 50` |
| yellow | 30-70% on either dimension |
| red | >70% on either OR `docker_status != running` |
| grey | no metrics available (e.g., service that doesn't expose container metrics) |

Category info preserved as the subtitle field (already present), not
as primary visual signal.

**T3 — Grafana Service Topology dashboard**
- Verify existing dashboard or create new one. JSON model in
  `config/grafana/dashboards/service-topology.json`.
- Uses Node Graph panel via `marcusolsson-json-datasource` pointed
  at topology-api `/api/topology`.
- Hint text in tooltip from `detail__purpose`. Refresh cadence ~30s.

**T4 — Health-and-utilization overlay panel**

Companion panel on same dashboard: per-service utilization
sparklines, restart-count counter, and docker status indicators.
Cross-references the Node Graph (clicking a node filters the
overlay panel).

**T5 — Drift gating**

topology-api's existing pre-commit lint stays. Add: if a service is
in NetBox but not in topology-api's response, fail. If a node has
no xindex `/service/{name}` match, fail (prevents drift between
ingested knowledge and rendered topology).

**Estimate:** 12 hours.

**Dependencies:** requires 17.F closed (so retired services are out
of topology-api before utilization wiring; otherwise wasted work).
No OpenProject dependency.

### 17.U — OPNsense DNS state audit + Unbound disable + retroactive incident review

Added 2026-05-01 post-237396b amend. Resolves the provisional state
captured in `docs/architecture-facts/opnsense-dns-authority.md` and
the open follow-ups in KI-009.

**Trigger.** Commit 237396b asserted "Dnsmasq is the active DNS
authority" based on probe evidence (Unbound host_overrides=0, Dnsmasq
hosts=6). Operator correction: probe data is consistent with the
operator-INTENDED architecture being Dnsmasq + Kea (and possibly
AdGuard Home), with Unbound's daemon being SUSPECTED unintended state.
Authority is an operator-intent question, not a probe-result question.

**Suspected current state (per `docs/architecture-facts/opnsense-dns-authority.md`).**
- Operator-intended authority: Dnsmasq + Kea (modern OPNsense 26.1
  default), NOT Unbound.
- Unbound's daemon is currently active on port 53 — likely enabled
  in a prior session by an earlier Claude/Aider/operator action that
  was never rolled back.
- 31 of 32 Caddy `.internal` sites resolve, but the records do NOT
  appear in either Unbound's host_overrides table or Dnsmasq's hosts
  table. Source of resolution is unknown.
- Per OPNsense 26.1 docs: when running Dnsmasq as resolver, "If you
  have Unbound DNS or any other DNS resolvers installed, make sure
  to disable them." Current state contradicts this guidance.
- Operator has seen this failure mode pre-Claude: two DNS services
  running simultaneously, port mapping breaks, requires full back-out
  of one to recover.

**Deliverables (T1-T9):**

**T1 — Operator-intent confirmation**
- Review OPNsense GUI Services list with operator (screenshot or
  side-by-side session)
- Confirm: Dnsmasq + Kea is the intended authority architecture
- Confirm: Unbound's running state is unintended (vs deliberately
  enabled for a forgotten reason)
- Document confirmation in revised architecture-facts file

**T2 — Identify source of currently-resolving `.internal` records**
- Probe Unbound's full config (`/api/unbound/settings/get`) for any
  custom_options, autogenerated entries, leases-derived entries,
  forwarder rules, or stub zones that could explain the 31 working
  records
- Probe Dnsmasq's full config for any wildcard/regex/synthetic-DNS
  entries beyond the 6 host records
- Check OPNsense System: Settings: General for domain auto-append
  rules
- Check Caddy `tls internal` directive — does it register records
  anywhere DNS-visible?
- Document source(s) found in revised architecture-facts file

**T3 — Migration plan (records currently served by Unbound → operator-intended authority)**
- Enumerate all currently-resolving records
- For each, confirm whether it should remain resolvable
- Add records that should remain to Dnsmasq Hosts (or wherever T1
  determines is operator-intended)
- Validate each migrated record resolves via the new authority
  BEFORE Unbound is disabled

**T4 — Unbound clean disable**
- Via OPNsense Services: Unbound DNS: General: uncheck Enable
- Apply changes
- Validate: port 53 still answers (now from Dnsmasq), all migrated
  records resolve, no records lost
- Validate: parity check (`scripts/check-repo-coherence.py
  caddy-dns-parity`) accurately reports state

**T5 — Re-audit and replace architecture-facts file**
- Replace `docs/architecture-facts/opnsense-dns-authority.md`
