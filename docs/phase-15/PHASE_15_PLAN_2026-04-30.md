### 1.1 Decision Frame — Orchestration Hub vs. Specialized Node

**The core question:** Does the Mac Studio M3 Ultra (96GB) become the new orchestration
hub (taking over the 62-container stack from the Mac Mini), or a specialized workload
node running a subset of services?

**Factors:**

| Factor | Argues for Hub | Argues for Specialized |
|---|---|---|
| RAM (96 GB) | 2× the Mini's 48 GB — comfortable headroom for full stack + growth | Overkill for orchestration; better spent on large models |
| M3 Ultra neural engine | Largest Apple Silicon neural engine; 76-core GPU | Same argument — ML inference, not container orchestration |
| Migration blast radius | Moving all 62 containers is high-risk, multi-day effort | Additive integration is incremental and reversible |
| Vault (critical single point) | Vault migration requires planned downtime + key ceremony | Vault stays on Mini; no key ceremony needed |
| Platform doctrine (portability-flagged) | Every decision is portability-flagged for Linux/Threadripper | Split architecture validates the multi-node portability story now |
| Phase 13 Increments 2B–7 | Already gated on external prereqs; delay compounds if migration also gates them | Parallel: Mini runs existing stack while Studio onboards specialized workloads |
| Operational risk | A failed migration could take down all 62 containers | Additive node: failure blast radius = Studio only |
| MacBook Pro M5 Block 3 | Block 3 was "MacBook Pro M5 parity: Ollama + LiteLLM + OpenWebUI + Headscale client" | Mac Studio supersedes MacBook as the compute node; Block 3 scope shifts |

**Recommendation: Mac Studio as Specialized Compute Node (not orchestration hub).**

Rationale:
1. The Mac Mini is already hardened, Vault-anchored, and running 62 containers cleanly. Migrating it to the Studio is weeks of migration risk for zero new capability.
2. The Mac Studio's value is its neural engine and 96 GB unified memory — ideal for large-model inference (70B+ models), not container orchestration. Putting Caddy, Plane, and NetBox on it wastes that advantage.
3. The MacBook Pro M5 Block 3 (Ollama parity node) is superseded: the Studio is a better fit for that role and more. Block 3 scope should be re-scoped to Studio.
4. An additive second node validates the multi-node architecture the platform has always been designed for (portability flags, per-host vault-configs, Headscale).
5. Vault stays on the Mini. No key ceremony, no downtime.

**The split:**

```
Mac Mini M4 Pro (192.168.10.145) — Control Plane (unchanged)
  All 62 existing containers: Vault, Caddy, Plane, NetBox, InvenTree,
  Grafana, Zabbix, observability stack, MCP layer, media stack,
  Home Assistant, platform services

Mac Studio M3 Ultra (TBD IP) — Compute Node
  AI/ML inference (large models):
    - Ollama (large models: llama3:70b, qwen2.5-coder:32b upgrade, etc.)
    - LiteLLM Gateway (Studio route addition; Mini route retained)
    - OpenWebUI (second instance or proxy to Studio Ollama)
    - AnythingLLM (can use Studio Ollama as backend)
    - OpenHands (compute-intensive; benefits from 96 GB)
  Development tooling:
    - Aider subagent workers (decomposer, implementer, reviewer)
    - Code execution sandbox (OpenHands or Daytona)
  Future:
    - GPU-accelerated inference (if eGPU or RTX passthrough)
    - Training workloads
```

### 1.2 What This Does NOT Change

- **Vault:** Stays on Mac Mini. Studio services get their own AppRoles via the existing Vault on .145.
- **Caddy:** Stays on Mac Mini. Studio services get `*.internal` routes proxied through Mini's Caddy. Or Studio runs its own Caddy for studio-only routes — see sequencing.
- **NetBox CMDB:** Stays on Mac Mini. Studio registered as a new host in NetBox.
- **Headscale:** Stays on Mac Mini. Studio becomes a Headscale client (like Block 3 planned for MacBook).
- **Regression probe:** Expands to include Studio health checks.
- **Secrets doctrine:** Studio services use Vault Agent sidecars pulling from Mini's Vault.

### 1.3 Networking Approach

Two options:

**Option A — Shared LAN + Caddy on Mini (simpler):**
Studio services bind to Studio LAN IP (e.g., 192.168.10.146). Mini's Caddy proxies
`studio.internal`, `ollama.internal`, etc. via upstream to Studio IP. All `.internal`
DNS resolution stays at OPNsense pointing to Mini's Caddy IP. No Headscale required
for LAN access.

**Option B — Headscale mesh (more robust):**
Studio joins Headscale. Services on Studio accessible via Tailscale IP across nodes.
Caddy on Mini can upstream to Headscale address. Enables remote access and multi-site
routing. Requires Headscale client setup on Studio (same as Block 3 plan for MacBook).

**Recommendation: Option A for initial integration.** Headscale mesh is Phase 15.C
(after Studio is stable). Reverting a Headscale misconfiguration is easier than
reverting a bad initial setup.

### 5.1 Block 4.H — Upgrade-Watcher Service (open, no prereqs)

Build a service that polls upstream registries for pinned image/binary release updates.

Input: `docker/*/docker-compose.yml` pinned image tags + `config/service-registry.yaml` binary versions.
Output: Plane issues / Grafana alerts / Slack (or Uptime Kuma webhook) when a new tag is available.

Implementation options (evaluate in order):
1. **Renovate Bot** running locally (container or cron) — zero custom code; reads compose files natively; opens Plane issues via webhook. Check `renovateapp/renovate` image size and ARM64 support.
2. **Diun** (Docker Image Update Notifier) — lighter than Renovate; ARM64 native; notifies via webhook. Outputs can be forwarded to Plane via existing webhook adapter.
3. **Custom script** (`scripts/upgrade-watcher.py`) — reads `docker inspect` + queries Docker Hub / GHCR API. Most control, most maintenance burden. Prefer option 1 or 2.

Effort: 4–7h (Renovate/Diun: 2–3h; custom: 5–7h). Apply ADR-A-013 fold gates.
