# Capability Audit — Zabbix

**Date:** 2026-05-01
**Auditor:** Claude session (operator-reviewed)
**Trigger:** Stack-level pattern-match recommendation suggested
"Zabbix is redundant; VictoriaMetrics covers this — retire it." This
audit is the D#20 evidence layer that overrides that recommendation.

## Section 2 — Probed capabilities (NOT documented capabilities)

Probes against the running zabbix-server via the API
(`http://localhost/api_jsonrpc.php`) on 2026-05-01.

### 2.1 — Item collection (the load-bearing capability)

**Probe:** `item.get` API call, count by type.
**Output excerpt:**

```
Total items:        19,072
By type:
  Dependent item:   11,174    (derived from other items)
  SNMP agent:        4,593    (network gear, OPNsense, switches)
  JMX agent:           510    (JVM-based services)
  Zabbix agent:        706    (host metrics from zabbix-agent
                                containers)
  HTTP agent:          ...    (smaller counts; not load-bearing)
  Other types:       ~2,089
```

The 4,593 SNMP + 510 JMX items are the load-bearing evidence — see
Section 3.

### 2.2 — Host coverage

**Probe:** `host.get` API call.
**Output excerpt:**

```
Total hosts:           55
```

Includes mac-mini, OPNsense, switches, JVM services, satellites.

### 2.3 — Trigger / alerting state

**Probe:** `trigger.get` API call (filter: enabled).
**Output excerpt:**

```
Active triggers:    7,198
```

These are the rules that fire alerts. Migrating off Zabbix means
re-authoring or replacing these rules in the destination tool's
alerting language — non-trivial.

### 2.4 — Prometheus exposition (for Grafana)

**Probe:** `curl http://zabbix-exporter:9224/metrics | head`.
**Output excerpt:**

```
zabbix_triggers_active{severity="warning"} 142
zabbix_triggers_active{severity="high"}    18
zabbix_hosts_available{status="available"} 53
...
```

`zabbix-exporter` (Phase 14 D-ZBX custom build) translates Zabbix
state into Prometheus metrics consumed by vmagent → VictoriaMetrics →
Grafana `zabbix-overview-p14` dashboard. **Status note (initial
probe):** container reports healthy and serves a successful response,
but `docker logs` shows continuous `Collect error: timed out` between
successful scrapes — the underlying `trigger.get`+`host.get` calls
exceed the exporter's 15s urlopen timeout under load. Bridge is
unreliable, not cleanly broken.

**Bridge decision (added in D-17-05, 2026-05-01): RETIRE the bridge.**
The exporter exposes only 2 metrics; Grafana already has a native
Zabbix data source; cross-stack PromQL over those 2 metrics is not
load-bearing. See
`docs/runbooks/observability-doctrine.md` §"Bridge between stacks"
for justification and migration plan. The Zabbix tool itself is
unaffected — only the Prometheus bridge component retires.

### 2.5 — Available but not exercised

- **Auto-discovery**: enabled but not heavily used; OPNsense and
  switches are statically registered.
- **Action scripts** (Zabbix's notification routing): minimally
  configured; alerts route via webhook to Grafana → ntfy. Not the
  load-bearing path, but available.
- **Inventory module**: not used (NetBox is the CMDB).

These are documented capabilities of Zabbix that this deployment
doesn't currently exercise. They are NOT counted as evidence for
KEEP — they would have to be re-built elsewhere only if we re-built
them anywhere.

## Section 3 — Stack-coverage check

For each load-bearing capability from Section 2:

### 3.1 — SNMP item collection (4,593 items)

- **What other tool in the stack covers this?** None.
  - VictoriaMetrics: no SNMP collector. vmagent scrapes Prometheus
    exposition, not SNMP.
  - Telegraf: not deployed. Adding it would mean a new tool to
    re-author 4,593 items in.
  - node-exporter: x86/ARM host metrics only; no SNMP.
- **Verdict for this capability:** UNIQUE to Zabbix in current stack.
  Migration would require deploying SNMP-Exporter (Prometheus) +
  re-authoring all 4,593 items as SNMP-Exporter generator-config
  YAML. Not zero work; not within Phase 17 scope.

### 3.2 — JMX item collection (510 items)

- **What other tool in the stack covers this?** None.
  - VictoriaMetrics: no JMX collection.
  - jmx_exporter (Prometheus side-car) would be the alternative, but
    it's per-JVM-process; Zabbix's JMX poller talks to one Java
    Gateway and pulls from many JVMs centrally — different topology.
- **Verdict for this capability:** UNIQUE to Zabbix in current stack.
  Same migration shape as SNMP — possible, not free, not warranted
  by current stack scale.

### 3.3 — Zabbix-agent host metrics (706 items)

- **What other tool in the stack covers this?** node-exporter (where
  deployed) and cAdvisor (container-level). This is the **one
  overlap.**
- **Cleaner where?** Prometheus side. node-exporter has a richer
  metric set for Linux/macOS hosts; vmagent already scrapes it; the
  data lands in the same VictoriaMetrics → Grafana path. Zabbix-agent
  carries 706 items here that duplicate what node-exporter exposes.
- **Verdict for this capability:** OVERLAPPING. Queue for
  consolidation — see Section 4 role-clarification + Section 5
  (which is N/A for the tool but applicable to this slice).

### 3.4 — Trigger/alert authorship (7,198 rules)

- **What other tool in the stack covers this?** Grafana alerting
  (already in use for Prometheus-side alerts).
- **Cleaner where?** Mixed. Grafana alerting is cleaner for
  PromQL-style threshold rules on time-series; Zabbix's expression
  language is cleaner for state-machine triggers, dependency
  hierarchies, and SNMP-trap-driven rules. The 7,198 rules are not
  homogeneous — some belong in each tool.
- **Verdict for this capability:** OWNERSHIP-SPLIT. Section 4
  documents which tool owns which rule type going forward.

### 3.5 — Prometheus exposition (zabbix-exporter)

- **What other tool covers this?** N/A — this is the bridge between
  Zabbix and the Prometheus-side stack. Its existence is what makes
  the role-clarification verdict feasible.

## Section 4 — Verdict

**KEEP WITH ROLE-CLARIFICATION**

Justification:
- Section 3.1 + 3.2 establish UNIQUE coverage (5,103 items have no
  cleaner home in the current stack).
- Section 3.3 establishes one consolidation opportunity (706 items).
- Section 3.4 establishes a co-existence pattern (Grafana alerting +
  Zabbix triggers, each owning what suits its language).

### Role clarification (the doctrine output of this audit)

Going forward, in this stack:

- **Zabbix owns:** SNMP collection (network gear, OPNsense,
  switches), JMX collection (JVM services), and trigger rules whose
  expression depends on SNMP state, JMX state, or Zabbix-native
  dependency hierarchies.
- **node-exporter + VictoriaMetrics owns:** host-OS metrics
  (CPU/memory/disk/network on Linux/macOS hosts), container metrics
  (cAdvisor), application Prometheus endpoints.
- **Grafana alerting owns:** threshold rules over Prometheus-side
  time-series (PromQL).
- **Bridge:** zabbix-exporter exposes Zabbix trigger/host state to
  the Prometheus side so Grafana dashboards have one pane of glass
  even though the collection planes are split.

### Queued cleanup

The 706 host-metric items in Zabbix that overlap node-exporter are
queued for retirement on the Zabbix side (not the whole tool — just
those items). Tracked as a follow-up; not in Phase 17 scope.

## Postscript — what this audit changed

Before the audit: pattern-matched recommendation was "retire
Zabbix; VictoriaMetrics covers this." Implementing that
recommendation would have silently dropped 5,103 SNMP + JMX items
with no replacement collector in the stack. The drop would have
become visible only when an alert that depended on those items
failed to fire — by which time the gap might have caused a real
incident.

After the audit: Zabbix is retained with an explicit role
(SNMP/JMX/state-machine triggers), VictoriaMetrics is retained with
an explicit role (host/container/app Prometheus metrics), and one
narrow overlap (706 items) is queued for cleanup. The stack is now
documented to be coherent rather than assumed-redundant.

This is what D#20 exists to enforce.
