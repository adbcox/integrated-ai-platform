## State at retirement

- **Container:** `sportarr` on Mac Mini, image
  `sportarr/sportarr:latest`, port `1867`, uptime 24 h, 0 restarts.
- **Indexer URL bug (root cause):** all 5 Prowlarr-backed indexers
  in `/config/sportarr.db` pointed to `http://localhost:9696/...`
  (sportarr's own loopback) instead of `http://prowlarr:9696/...`
  (Prowlarr container reachable on `control-center-net`). Every
  fetch returned `Connection refused (localhost:9696)`.
- **Channel-mapping gap:** 3 monitored events with "No Channel" →
  DVR Auto-Scheduler couldn't schedule even if downloads worked.
- **Failure volume:** `docker logs sportarr 2>&1 | grep -c "URL
  Fetching failed"` → **6,769** failures across the log window.
- **Resource burn:** 394.6 MiB resident, 35 PIDs, 948 MB net rx /
  24 MB tx in 24 h — all wasted retry traffic.

## Restoration steps

If a future operator decides to revive sportarr:

1. **Uncomment the `sportarr:` block** in
   `/Users/admin/control-center-stack/stacks/arr-stack/docker-compose.yml`
   (remove the leading `# ` from each line of the block; leave the
   "RETIRED" header comment in place as historical context).
2. **Fix the indexer URLs FIRST** before bringing the container up.
   **SCHEMA CORRECTION (2026-05-03 D-17-36 WP-08):** the column is
   `Url`, not `BaseUrl`. Original retirement-era record cited the
   wrong column name (likely from a Prowlarr export not from
   `sportarr.db`). The correct UPDATE:
   - Mount the volume read-write into a temporary alpine container
     and `sqlite3 /config/sportarr.db "UPDATE Indexers SET Url =
     REPLACE(Url, 'localhost:9696', 'prowlarr:9696');"`, OR
   - Bring sportarr up with the bug, fix via the web UI at
     `:1867` (more error-prone but doesn't require sqlite-fluency).

2.5. **Refresh indexer ApiKey against live Prowlarr** before
   verifying. **NEW STEP (2026-05-03 D-17-36 WP-04):** retirement
   record did not mention ApiKey at all. On unpark, all 5 indexers
   had stale ApiKey that no longer matched live Prowlarr's key.
   Url fix landed but every fetch returned 401 until:
   ```
   PROWLARR_KEY=$(docker exec prowlarr cat /config/config.xml \
     | grep -oE '<ApiKey>[^<]+' | sed 's/<ApiKey>//')
   sqlite3 /config/sportarr.db \
     "UPDATE Indexers SET ApiKey = '$PROWLARR_KEY';"
   ```
   Sportarr auto-creates duplicate Indexer rows (with/without
   trailing slash on URL) when re-fetched against authenticated
   Prowlarr; sweep ApiKey across ALL rows, not just the original 5.
3. **Bring sportarr up:**
   `docker compose -f /Users/admin/control-center-stack/stacks/arr-stack/docker-compose.yml up -d sportarr`
4. **Verify:** `docker logs sportarr --tail 50` should show
   `Fetched N total releases from 5 RSS feeds` with N>0 (or at
   least no "Connection refused" errors).
5. **Then** address the channel-mapping issue — the 3 monitored
   events with "No Channel" need operator-knowledge mapping work
   in the sportarr UI before DVR Auto-Scheduler will schedule
   anything.

## Before restoring, ask

  downloads and the silence around them is strong evidence the
  capability isn't being missed. If you're considering restoration,

  through purpose-built services (Sportbots, MissingScreen, etc.)
  with different operational profiles. Don't restore sportarr
  reflexively just because the compose comment is still there.

  fails silently for 6,769 cycles before anyone notices isn't a
  tool the operator is engaged with. Restoring it without a
  changed engagement plan likely lands you back in the same state
  in another quarter.

If the answer to "do you want this" is genuinely yes after asking
those questions, restore. Otherwise, leave the comment in place;
the volume is safe; future-you can revisit again.

## Reversal record (2026-05-03, D-17-36)

**Decision:** PARK-RETIRE reversed. Sportarr unpark for F1 coverage
+ general sports-DVR substrate.

**Trigger.** Operator F1 coverage requirement (qualifying / sprint /
full race). Initial pivot to "Sonarr-for-F1 instead" was reversed
after pre-flight inspection surfaced empirical evidence that the
Sonarr workaround had already been tried for ~7 weeks and was failing.

### Empirical evidence (the dispositive datapoint)

Operator added **Formula 1 (TVDB 387219) to Sonarr on 2026-03-15** —
six weeks *before* this retirement was authored — as a workaround
because Sportarr wasn't working. State as of 2026-05-03 inspection:

```
series_id      = 185
title          = Formula 1
tvdbId         = 387219
seriesType     = standard
useSceneNumbering = false
qualityProfileId  = 1 (Any)
path           = /data/media/tv/Formula 1
monitored      = true (season 2026 only; 2022-2025 unmonitored)
season 2026    = 9 episodes registered, 5 episode files
               = 13.08 GiB on disk
               = 55.6% completion
```

**The 55.6% completion rate over 7 weeks of Sonarr operation is
the dispositive evidence** that Sonarr is the wrong tool for sports
content. Sonarr's TV-show parser does not match the F1 release
naming conventions reliably (post-broadcast WEB-DL drops with
session-specific tags); episodes register but releases never grab.
This is exactly the capability gap Sportarr was built to fill.

### Gate answers (responding to §"Before restoring, ask")

The 2026-05-01 retirement record posed three gates that needed
operator answers before restoration. Answers as of 2026-05-03:

**Gate 1 — "Do you actually want live-sports DVR?"**

YES. F1 is the canonical use case (qualifying + sprint + race per
round, ~24 rounds/year). The 2026-05-01 retirement was authored
before this requirement crystallized; the 6,769 historical failures
reflected zero engaged use case at that time, not absence of latent
need. With F1 as a concrete in-use case, sports-DVR becomes the
goal, not just F1-specific file pulls.

**Gate 2 — "Have alternatives been considered?"**

YES, EMPIRICALLY. The Sonarr workaround was tried for 7 weeks
between 2026-03-15 and 2026-05-03. Result: 55.6% completion, files
that *do* land require manual intervention to organize, season
mapping breaks (TVDB uses season=year + episode=session-in-year
which Sonarr's parser doesn't reliably match against release
names like `Formula.1.2026.Round.05.Miami.Race.1080p.F1TV.WEB-DL`).
The workaround has been falsified by experiment, not just by
prediction. Other alternatives mentioned in the retirement record
(Sportbots, MissingScreen) are unevaluated; Sportarr is operator's
chosen tool given the existing volume + indexer config + arr-stack
pattern fit.

**Gate 3 — "What's different this time? (signal value of long-
running silent breakage)"**

Three structural changes since 2026-05-01 prevent the same silent-
breakage recurrence:

1. **Operator engagement is now active.** F1 is being actively
   consumed — silent failure would surface within one race weekend
   (24 hours, not 6,769 retry cycles).

2. **Audit-discipline doctrine.** D-17-32 stack integration audit
   established the doctrine that "container `(healthy)` is not
   `(integration working)`" — Sportarr post-unpark will be
   evaluated against integration-path checks, not container
   liveness alone.

3. **F5/F8 integration-health-check family.** Gap F5 in
   `docs/_audit/integrated-stack-gaps-2026-05-03.md` (canonical
   instance: 2026-05-03 Sonarr/QNAP SMB mount break that went
   undetected) and Gap F8 (QNAP SMB mount persistence work blocked
   behind Finding Y) are the framework for end-to-end probes that
   would catch a recurring Sportarr failure — `Fetched N total
   releases` log assertion + Plex import-success probe + per-round
   coverage check. The family hasn't been authored as a deliverable
   yet, but the doctrine is in place; Sportarr post-unpark will be
   one of the F5-B (media-stack tier) target flows.
